MySQL数据库设计规范

数据库 MySQL 2018/04/26

本文于385天之前发表，文中内容可能已经过时。

数据库命名规范

  1. 所有数据库对象名称必须小写
  2. 命名要做到见名识意，禁止使用MySQL保留关键字
  3. 临时表以tmp_开头，备份表以bak_开头并以时间戳结尾
  4. 所有存储相同数据的列名和列类型必须一致

数据库基本设计规范

  1. 所有表必须使用Innodb存储引擎
    1. 支持事务，行级锁，更好的恢复性，高并发下性能更好
    2. show engines;
  2. 数据库和表的字符集统一使用UTF-8
    1. 避免由于字符集转换产生乱码
    2. 汉字占3个字节，ASCII码占1个字节
  3. 所有表和字段必须添加注释
  4. 尽量控制单表数据量的大小，建议控制在500万行以内
    1. 可以用历史数据归档，分库分表等手段来控制数据量大小
  5. 谨慎使用MySQL分区表
    1. 分区表在物理上表现为多个文件，在逻辑上表现为一个表
    2. 跨分区查询效率会低
    3. 建议采用物理分表的方式管理大数据
  6. 尽量做到冷热数据分离，减小表宽度
    1. 减少磁盘I/O，保证热数据的内存缓存命中率
    2. 更有效的利用缓存，避免读入无用的冷数据
    3. 经常一起使用的列放到一个表中
  7. 禁止在表中建立预留字段
  8. 禁止在数据库中存储图片，文件等二进制数据
  9. 禁止在线上做数据库压力测试
  10. 禁止从开发环境、测试环境直连生产环境数据库

数据库索引设计规范

  1. 限制每张表上的索引数量，建议单张表索引不超过5个
  2. 禁止给表中的每一列都建立单独的索引
  3. 每张表都必须有一个主键
    1. 不使用更新频繁的列作为主键
    2. 不使用多列主键
    3. 不使用UUID，MD5，HASH，字符串作为主键
    4. 主键建议使用自增id值
  4. WHERE从句、ORDER BY、GROUP BY、DISTINCT、多表JOIN中的列优先设置索引
  5. 设置索引列的顺序
    1. 区分度最高的列放在联合索引的最左侧
    2. 尽量把字段长度小的列放在联合索引的最左侧
    3. 使用最频繁的列放到联合索引的左侧
  6. 避免建立冗余索引和重复索引
    1. primary key(id)、index(id)、unique index(id)
    2. index(a,b,c)、index(a,b)、index(a)
  7. 对于频繁的查询优先考虑使用覆盖索引
    1. 避免Innodb表进行索引的二次查找
    2. 可以把随机I/O变为顺序I/O加快查询效率
  8. 尽量避免使用外键
    1. 外键可用于保证数据的参照完整性，但建议在业务端实现
    2. 外键会影响父表和子表的写操作从而降低性能

数据库字段设计规范

  1. 优先选择符合存储需要的最小的数据类型
    1. 将字符串转化为数字类型存储，INET_ATON(‘255.255.255.255’)，INET_NTOA(4294967295)
  2. 对于非负整数应采用无符号整型进行存储
  3. VARCHAR(N)中的N指定要明确，过大的长度会消耗更多的内存
  4. 避免使用TEXT、BLOB数据类型
    1. 建议把BLOB或者TEXT分离到单独的扩展表中
    2. TEXT或BLOB类型只能使用前缀索引
  5. 避免使用ENUM数据类型
  6. 尽可能把所有列定义为NOT NULL
    1. 索引NULL列需要额外的空间来保存
    2. 进行比较和计算要对NULL值做特别的处理
  7. 不要用字符串存储日期类型的数据
    1. 无法用日期函数进行计算和比较
    2. 用字符串存储日期要占更多的空间
  8. 使用TIMESTAMP或DATETIME类型存储时间
    1. TIMESTAMP占用4字节和INT相同，但比INT可读性高
    2. 超出TIMESTAMP取值范围的时候用DATETIME
    3. TIMESTAMP 1970～01～01 ～ 2038-01-19
  9. 财务相关的金额类数据，必须使用decimal类型

SQL开发规范

  1. 建议使用预编译语句进行数据库操作
  2. 避免数据类型的隐式转换
    1. select phone from user where id = ‘1’
  3. 充分利用表上已经存在的索引
    1. 避免使用双%号的查询条件
    2. 使用left join 或 not exists 优化not in操作
  4. 禁止跨库查询
  5. 禁止使用SELECT *，必须使用SELECT <字段列表> 查询
  6. 禁止使用不含字段列表的INSERT语句
  7. 避免使用子查询，可以把子查询优化为join操作
  8. 避免使用JOIN关联太多的表，建议不超过5个
  9. 合并多个相同操作到一起，减少同数据库交互次数
  10. 使用in代替or，in的值不要超过500，in操作可以利用索引
  11. 禁止使用order by rand() 进行随机排序
  12. WHERE从句中禁止队列进行函数转换和计算
    1. where date(time) = ‘20160901’
    2. where createtime >= ‘20160901’ and createtime < ‘20160902’
  13. 在明显不会有重复值时使用UNION ALL，而不是UNION
  14. 拆分复杂的大SQL为多个小SQL
    1. 一个SQL只能使用一个CPU进行计算

数据库操作规范

  1. 超过100万行的批量写操作，要分批多次进行操作
    1. 大批量操作可能会造成严重的主从延迟
    2. binlog日志为ROW格式会产生大量的日志
    3. 避免产生大事务操作
  2. 对大表数据结构的修改要谨慎，会造成严重的锁表操作
    1. 使用pt-online-schema-change修改表结构
  3. 禁止为程序使用的账号赋予super权限
  4. 程序使用的账号原则上不准有drop权限
